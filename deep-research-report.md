# Wildfire Spread Predictor – Architecture and Training Pipeline

**Overview:** We build a real-time wildfire spread simulator powered by a deep learning model. Live weather (wind, humidity, temperature) and static fuel/terrain maps are fed into a trained U-Net, which predicts fire ignition probabilities. The frontend displays the spread on an Esri/Leaflet map with alerts. An LLM serves as an AI assistant for explanation and decision support.

## 1. Data Integration Engine  
- **Live Weather Feed:** Fetch wind speed, wind direction, humidity, and temperature via the free, no-key Open-Meteo API. For example, Open-Meteo’s JSON API returns current wind and humidity data at any lat/lon with a simple HTTP GET. Each fetch is timestamped and logged.  
- **Fuel/Terrain Data:** Load a static raster grid of vegetation density and topography. In practice, we would use real land-cover and elevation datasets (e.g. Copernicus/NASA land cover and SRTM) pre-aligned to the map grid. In prototype, a synthetic grid marks cells as forest, grass, river, road, or town. Each cell’s type is color-coded on the map (green=forest, olive=grass, blue=water, tan=town, gray=road/firebreak).  
- **Normalization & Logging:** Before feeding to the model, numeric inputs are normalized (e.g. min-max scaling) for consistency. All environmental readings (wind, humidity, etc.) are logged with timestamps in a table (for offline analysis). This satisfies the need to “timestamp and log every reading” as a persistent record.  

## 2. Wildfire Spread Simulation Model  
- **Dataset & Features:** We train on the **Next-Day Wildfire Spread** dataset (a large Kaggle/IEEE TGRS dataset). It includes multi-channel inputs: current-day fire mask, topography, vegetation index, wind vectors, weather, drought index, etc., aligned over 2D regions. (In total, 10–12 channels such as wind speed/direction, humidity, drought, vegetation, elevation, etc., as used by Fitzgerald et al..) These modalities drive where fire can propagate next day.  
- **U-Net Architecture:** We implement a 2D U-Net (e.g. via PyTorch’s [segmentation_models_pytorch](https://github.com/qubvel/segmentation_models.pytorch)) with a ResNet34 encoder pretrained on ImageNet. The model takes current conditions (fuel/terrain + weather + fire mask) as input and outputs a next-day fire mask or spread-probability map. During training, we use a binary cross-entropy (or focal) loss on the binary fire extent, optionally weighting for the class imbalance (since >97% pixels are non-fire). In recent work, including pretraining and using focal loss improved performance substantially.  
- **Training Details:** We follow best practices from recent literature. For example, Gérard et al. (2024) trained a Res18-UNet on these data using segmentation_models_pytorch with `encoder_weights='imagenet'`. We use AdamW optimizer with a fixed learning rate, batch size ~64, and train for ~10,000–15,000 iterations (roughly 10–15 epochs on shuffled data). Pretraining the encoder and using focal loss (γ=2, α = inverse class frequency) boosted accuracy. We monitor precision/recall or average precision (AP) on a validation split each epoch, saving the best model weights. The final model file (PyTorch `.pt`) is the key artifact passed to the live app.  

## 3. Frontend Map & UI  
- **Interactive Map:** The left pane shows a Leaflet map using Esri’s World Imagery basemap (1m satellite imagery, no API key required). A semi-transparent grid overlay divides the terrain into square cells. Each cell is color-coded: green for forest, olive for dry grassland, blue for water, tan for town, and dark gray for roads/firebreaks. A legend below the map explains these colors. This provides a realistic backdrop for the simulated fire.  
- **Ignition & Animation:** Clicking on a land cell “ignites” a fire there. The fire then spreads cell-by-cell: burning cells glow orange, then turn to black/ash after burning out. Spread is driven by the model’s output probabilities and wind alignment. We animate the fire propagation in real time (e.g. one tick per minute of real time, adjustable). Each tick queries the model’s prediction (a spread-probability grid) to determine which adjacent cells ignite next, factoring in wind direction (carry fire faster) and fuel.  
- **Control Panel (Right Side):** Users can select a region (drop-down) and choose “Live data” vs “Demo data”. In Live mode, current conditions from Open-Meteo are shown (e.g. “Wind: 12 km/h SE, Humidity: 34%, Temp: 29°C”). “Demo data” uses cached or static conditions to avoid network issues. Sliders and compass buttons allow manually overriding wind speed/direction and humidity, for “what-if” tests. A timestamp updates with each data fetch. Run/Pause/Reset/Ignite buttons control the simulation. Stats display elapsed time, % area burned, number of active fire-front cells, and cells already burnt.  
- **Evacuation Alerts:** A town cell is marked on the map. If the expanding fire perimeter comes within a threshold distance of the town, a red flashing alert appears in the panel (“Fire approaching Town!”). At that moment, an evacuation direction is drawn (e.g. an arrow pointing away along the nearest safe path) and a suggested route line is overlaid on the map. This route is computed simply: for example, pick the nearest road/firebreak that leads away from the fire’s centroid. (More advanced GIS routing could use the full road network if time permits.)  

## 4. Evacuation & Resource Planning  
- **Alerts & Routes:** The system proactively alerts before the fire reaches populated zones. When a high-risk approach is detected, the alert panel recommends evacuation. We draw a path (on the map) leading from the threatened town toward safety (away from wind-borne embers). For example, one might use Dijkstra on the grid: designate cells farthest from current fire front as “safe exits” and compute the shortest road-connected path. The UI highlights this path in a distinct color (e.g. yellow arrow) and labels it “Evacuation Route.”  
- **Resource Deployment:** In the panel we include a simple resource allocator. For instance, we could show “Deploy Unit 3 to Sector B” where sectors are predefined zones (e.g. quadrants of the map). A rule-based algorithm assigns the nearest available firefighting unit to the highest-risk sector: compute distance from each station to each fire front, pick minimal time-to-arrival. This can be done greedily each tick. (More sophisticated allocation could optimize coverage, but a simple heuristic suffices for MVP.)  
- **LLM Assistance in Planning:** An integrated LLM can augment evacuation/resource suggestions. For example, if the system detects a major infrastructure at risk, the LLM (with relevant data) could generate a text alert like: “Consider evacuating Highway 65, as fire is moving northeast rapidly.” The LLM can draw on built-in knowledge (e.g. typical fire behavior) and real-time data to advise. In essence, we envision a chat-style “Wildfire Assistant” that operators can query (see below).

## 5. System Architecture  

 *Image: **System Architecture Diagram.** Data flows from weather APIs and static maps into the backend model, whose outputs feed the frontend UI. (Diagram not shown here.)*  

- **Layers:** The system has three main layers:
  - **Data Layer:** Handles external data. This includes the Open-Meteo API for current weather (wind, humidity, etc.) and preloaded GIS layers for vegetation and terrain. All data is geo-referenced to the grid.  
  - **Model Backend:** A Python/Flask (or FastAPI) service loads the pre-trained PyTorch U-Net. On each simulation tick, it receives the current grid state (fire mask, wind vector, humidity, etc.), normalizes them, and runs the model to get a spread-probability map. The backend also logs conditions and triggers alerts.  
  - **Frontend UI:** An HTML/JavaScript (e.g. React or simple JS) app using Leaflet displays the map and grid. It polls or listens to the backend for updates (new burned cells, stats) and animates them. Controls (sliders, dropdowns) send commands (e.g. change wind) back to the backend to affect the next inference.  
- **Data Flow:** On “Ignite,” the frontend sends an initial fire mask to the backend. Then every tick: the backend queries Open-Meteo (if Live mode) or uses cached values, passes data into the model, and returns the new fire mask subset to ignite. The frontend then updates the colored cells. Alerts and LLM outputs are pushed from backend to frontend panels.  
- **Technology Stack:**  
  - **Map & UI:** Leaflet.js with Esri World Imagery basemap (no API key) for satellite view; HTML/CSS/JS or React for controls.  
  - **Weather API:** Open-Meteo for wind/humidity/temperature (free, no-key).  
  - **Model:** PyTorch with `segmentation_models_pytorch` (UNet, ResNet34 encoder) for inference. Model served via Python (Flask/FastAPI).  
  - **Data:** Next-Day Wildfire Spread dataset (Kaggle/Google Research) for training; optionally a second dataset like WildfireSpreadTS for time-series info.  
  - **Integration/Logging:** Python backend scripts to fetch API data, normalize, log to files or in-memory list, and serve JSON to frontend.

## 6. Model Training Pipeline  

1. **Data Download & Preprocessing:** Obtain the *Next-Day Wildfire Spread* dataset (e.g. via Kaggle API). This yields paired images: current fire mask and input features (12 channels) along with next-day fire mask. Resize or tile these to our grid resolution (e.g. 64×64). Normalize each channel: e.g. scale vegetation/temperature/humidity to [0,1]. Split into train/validation sets (e.g. by fire event or random).  
2. **Model Definition:** Using `segmentation_models_pytorch`, define a UNet model: encoder=ResNet34 (pretrained), decoder upsampling to original size. Set `encoder_weights='imagenet'` to leverage transfer learning. Output layer with sigmoid activation gives per-pixel burn probability.  
3. **Training Loop:** Use PyTorch Lightning or custom loop. Optimizer = AdamW (weight decay). Learning rate ~1e-4 (tuned via a small grid search if time). Loss = Focal Loss (γ=2) + BCE, or straight BCE as baseline. Batch size 64. Train for ~10–15 epochs (or fixed iterations, ~10k). After each epoch, evaluate on the val set using metrics (precision, recall, AP, F1). Early-stop or pick best epoch by highest validation AP or F1.  
4. **Save Weights:** The best-performing model (e.g. lowest val loss) is saved to disk. This `.pt` file is loaded by the live app. Store also any normalization parameters used.  
5. **Testing (Optional):** Compute final accuracy metrics. Gérard et al. achieved ~0.45 AP with a Res18-UNet on this task. We should ensure the model is better than a simple heuristic.  

## 7. LLM Integration for Decision Support  

- **Conversational Assistant:** Inspired by recent research, we can embed a Chatbot-style interface (“Wildfire Assistant”). This LLM (e.g. GPT-4 or Claude) is fine-tuned or prompted with domain context. It can answer user queries in natural language (e.g. “Why is the fire spreading faster now?” or “What risk does the town face?”). The LLM uses retrieval-augmented generation (RAG): it has access to local data (current fire status, weather, map features) and domain text (e.g. fire behavior rules). When a user asks, the LLM retrieves relevant data (e.g. “current wind is NE at 20 km/h and vegetation is dense”) and explains effects. This follows the paradigm of “multi-agent frameworks” where specialized LLM agents handle sub-tasks like simulation analysis or planning.  
- **Alert/Planning RAG:** When the fire threatens a zone, the backend can call the LLM to generate a detailed alert message. For example: *“The model predicts a 85% chance fire will reach Highway 12 in 30 min. Strong northeast winds will carry embers toward the town’s west side. Recommend evacuating along River Road (southbound).”* The LLM’s response is grounded in real-time inputs and data. In practice, this involves feeding the LLM a prompt like: “Fire front is at (x,y) moving N at 25 km/h; town is 5 km north of fire. Suggest actions.” RAG ensures it uses actual numbers rather than hallucinations. This matches concepts from the literature: e.g. “conversational AI assistants built on LLM+RAG could empower responders with data-driven advice”.  
- **Interactive Analytics:** The LLM can also generate human-readable summaries of the simulation. For instance, at each step it could be asked “Summarize the situation.” The LLM would consult the current burned area, front shape, wind, etc., and output something like: *“After 15 minutes, 12% of the area is burned. The fireline stretches 3 km northeast-southwest, driven by strong southerly winds. Probability of reaching the northern town has risen above 70%.”* This leverages its natural language ability to explain model outputs. Systems like WildfireGPT have demonstrated combining hazard data with LLMs to produce tailored risk insights.  
- **Evacuation Scenario Simulation:** Advanced: we could simulate human behavior using LLM-driven agents, as done in recent studies. For example, a user could ask the Assistant: “If the entire northern sector evacuates, how will traffic flow affect safety?” The LLM could run a quick CA simulation under that scenario (or recall similar patterns) and advise. Early work shows LLM agents can mimic crowd evacuation decisions in a grid-world fire scenario, though this is beyond MVP.  
- **Dialogue & Personalization:** Like multi-agent risk tools in research, we can make the Assistant user-centered. It could prompt for user role (fire chief, resident, planner), location interest, etc., then tailor responses. A profile (e.g. “emergency manager in zone C”) could guide which data and tone to use. In sum, the LLM acts as an on-demand advisor, translating raw model outputs and hazard data into actionable language (thanks to retrieval of documents, model results, maps).

## 8. References to Prior Work and Data  
- **Wildfire Datasets:** The “Next-Day Wildfire Spread” dataset was explicitly designed for this problem. It aggregates ~decade of satellite fire data with weather/topography. We use it as the training corpus.  
- **Deep Models:** Previous studies (e.g. Fitzgerald et al. 2023) used U-Net variants with attention on 12-channel data and achieved state-of-art spread prediction. We similarly adopt a UNet with ResNet encoder (pretrained) and expect comparable accuracy. Improved strategies (pretraining, focal loss) have been shown to boost AP by 30–40% over baseline.  
- **LLM + Simulation:** The idea of using LLMs for wildfire decision support is gaining traction. Recent work suggests LLMs (with RAG) can coordinate fire spread models and generate planning advice. In particular, WildfireGPT integrates multiple data layers (weather, fuel, demographics) and provides chat-based risk analysis with maps. Our assistant will be a lightweight version of this: using the trained model and live data to answer Q&A and recommend actions.  
- **Evacuation Modeling:** Studies have shown LLMs like GPT-4 can drive evacuation simulations in cellular automata environments. While we’re not training LLM agents, this shows LLM reasoning can align with fire dynamics. We will rely on simpler rule-based routing and use the LLM for explanation rather than internal agent movement.  

## 9. Summary of Key Components  
- **Map UI:** Leaflet + Esri imagery, overlaid with fuel cells. Real satellite backdrop gives realism (as used by OSM and Crisis Mapping teams).  
- **Model Inference:** PyTorch UNet, outputs per-cell burn probability each tick. Trained on satellite/fire datasets.  
- **Data Feeds:** Open-Meteo for weather (wind/humidity); static vegetation/elevation grid from remote-sensing sources.  
- **Backend:** Python service for pulling data, running model, sending updates.  
- **Frontend Controls:** React/JS toggles and sliders modify environmental inputs; simulation controls (ignite/pause/reset).  
- **Alerts & Planning:** Automatic detection of danger to towns triggers UI alerts and draws evacuation arrows. A panel suggests where to send firefighting units.  
- **LLM Assistant:** A GPT-like model fine-tuned for wildfire context, integrated via RAG to answer queries and draft advisories based on live data.  

Each piece maps to the project goals: real-time AI-driven spread (vs. hand-coded fire spread), integrated data feeds, and decision-support outputs (alerts, routes, explanations). By following this architecture and training process, we ensure the MVP has a working AI model at its core, hooks for live data, and an intelligent interface (both GUI and LLM-based) for emergency management.
